# Documento de Arquitectura: Plataforma VivaEventos

## 1. Introducción  
Este documento describe la arquitectura inicial del sistema de gestión de eventos y venta de boletas digitales de la plataforma **VivaEventos**, incluyendo los requisitos funcionales, requisitos de calidad y restricciones que deben considerarse durante el diseño e implementación del software.

El objetivo del sistema es permitir la publicación de eventos, la venta de boletas digitales con código QR, la integración con pasarelas de pago, la validación de acceso a los eventos y el monitoreo de ventas en tiempo real.

## 4. Restricciones del Sistema  
Las restricciones establecen **limitaciones** en la arquitectura del sistema, ya sean tecnológicas, de negocio, regulatorias o de infraestructura.

### **Lista de Restricciones**                                                                                                                                                          
| ID       | Tipo de Restricción              | Restricción                                                                                                                                                | Evidencia (cita del relato)                                                                                       |
| -------- | -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **R-01** | **Tiempo / Proyecto**            | El backend del sistema debe desarrollarse y entregarse como un **MVP funcional en un plazo máximo de 13 semanas**.                                         | “Yo necesito que ustedes construyan el backend del MVP en **13 semanas**.”                                        |
| **R-02** | **Arquitectura / Tecnológica**   | El sistema debe diseñarse utilizando una **arquitectura basada en microservicios**, evitando soluciones monolíticas.                                       | “No quiero un **sistema todo en uno**… necesito que esté **separado en servicios con responsabilidades claras**.” |
| **R-03** | **Arquitectura / Integración**   | Cada microservicio debe manejar su propia lógica y **no debe acceder directamente a la base de datos de otros servicios**.                                 | “No quiero que un servicio **toque la base de datos de otro como atajo**.”                                        |
| **R-04** | **Integración externa**          | El sistema debe integrarse con una **pasarela de pagos real en modo sandbox**, evitando simulaciones ficticias.                                            | “No quiero simulaciones ‘de mentiras’; necesito por lo menos **un sandbox real**…”                                |
| **R-05** | **Infraestructura / Despliegue** | La aplicación debe poder ejecutarse en un entorno reproducible mediante **contenedores**, facilitando su despliegue y pruebas por otros equipos.           | “Idealmente **con contenedores** para que el equipo de pruebas lo pueda levantar fácil.”                          |
| **R-06** | **Alcance / Producto**           | El sistema debe limitarse a las **funcionalidades esenciales del dominio de venta de eventos**, evitando características avanzadas fuera del MVP.          | “No estamos construyendo la NASA… No necesito motor de recomendaciones…”                                          |
| **R-07** | **Rendimiento / Escalabilidad**  | El sistema debe soportar **picos de miles de usuarios concurrentes** durante la apertura de ventas sin degradación crítica del servicio.                   | “Se nos puede meter mucha gente… he visto **picos de miles intentando comprar**…”                                 |
| **R-08** | **Negocio / Consistencia**       | El sistema **no debe permitir la venta de más boletas que el aforo disponible** bajo ninguna circunstancia.                                                | “No puedo permitir que se vendan más boletas de las que hay…”                                                     |
| **R-09** | **Integración / Pagos**          | El sistema debe manejar **estados de pago asincrónicos** definidos por la pasarela, incluyendo pagos pendientes, fallidos y confirmaciones posteriores.    | “Es normal que el pago quede en estado intermedio y llegue confirmación después…”                                 |
| **R-10** | **Infraestructura / Operación**  | El sistema debe ser capaz de **operar bajo condiciones de conectividad limitada** durante la validación de boletas en el ingreso a eventos.                | “El personal de logística está con celular y señal regular…”                                                      |
| **R-11** | **Seguridad / Regulación**       | El sistema debe implementar **control de acceso basado en roles y manejo seguro de credenciales y secretos**, evitando exposición de información sensible. | “No puedo exponer llaves… necesito roles, permisos y control de acceso.”                                          |

>  **Tipos de restricciones:**  
> - **Tecnológicas:** Lenguajes, frameworks o herramientas que deben utilizarse.
